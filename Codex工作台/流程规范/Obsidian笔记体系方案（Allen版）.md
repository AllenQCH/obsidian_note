# Obsidian笔记体系方案（Allen版）

## 摘要

这套方案不是追求最花哨的 Obsidian 玩法，而是为了解决当前主库“素材、项目、知识、索引混写”导致的混乱问题。

后续统一采用 **PARA-lite 目录分层 + MOC 主题入口 + Evergreen 知识沉淀 + Raw Inbox 缓冲区** 的组合方式。

核心原则只有一句话：**先判断这篇笔记属于哪一类，再决定写法和位置。**

## 核心内容

### 1. 推荐的长期方案

#### 目录层采用 PARA-lite

不强行推翻现有目录，而是在现有主库上逐步靠拢下面这套分层：

- `00 Inbox/`：临时记录、待整理摘录、还没判断归属的内容
- `01 Projects/`：阶段性推进事项、任务拆解、项目上下文、复盘
- `02 Areas/`：长期主题，例如 AI、投资、写作、自动化、知识管理
- `03 Resources/`：外部资料、课程摘录、网页剪藏、参考链接、素材仓
- `99 Archive/`：旧方案、完成项目、过期素材、历史导入内容

考虑到当前主库已经有既存结构，因此采用 **映射迁移**，而不是一次性大搬家：

- `ai-related/` 可视为 `02 Areas/AI`
- `项目相关/` 可视为 `01 Projects`
- `Codex工作台/` 可视为 `02 Areas/自动化工作台`
- `ai-related/OneNote恢复/` 应逐步收敛进 `03 Resources/` 或 `99 Archive/`
- 零散网页残留、导入附件、未整理草稿优先进入 `00 Inbox/`

#### 笔记内容采用 MOC + Evergreen

- **MOC / 索引页**：负责导航，不追求讲细节
- **Evergreen / 知识页**：只讲一个明确问题，用自己的话沉淀
- **项目页**：只服务推进，不强求知识化
- **素材页**：允许粗糙，但必须显式标记为 raw

这比全库统一成一种写法更适合 Allen 当前的使用方式。

### 2. 四类笔记的判断规则

#### A. 索引型笔记（Hub / MOC）

适用于：
- 一个主题下面已经有多篇笔记
- 需要一个总览入口
- 想降低“知道写过但找不到”的问题

判断问题：
- 这篇笔记是不是主要负责“带路”？

如果是，就写成索引型。

典型标题：
- `LLM总览`
- `AI工具索引`
- `投资研究地图`
- `自动化工作台`

#### B. 知识型笔记（Evergreen）

适用于：
- 一个概念、一种方法、一个判断
- 以后会反复复用
- 需要形成自己的表达而不是搬运原文

判断问题：
- 这篇笔记能不能一句话说清“它到底回答了什么问题”？

如果能，就写成知识型。

典型标题：
- `MCP解决的核心问题`
- `RAG为什么需要Rerank`
- `Claude Code 与 Codex 的差异`

#### C. 项目型笔记

适用于：
- 有明确目标和推进状态
- 与时间、任务、上下文强相关
- 更关注下一步行动，而不是知识优雅

判断问题：
- 这篇笔记是不是为了推动某件事继续往前走？

如果是，就写成项目型。

#### D. 素材型笔记（Raw / Resource）

适用于：
- 剪藏文章
- 临时摘录
- 会议随手记
- 尚未内化的内容

判断问题：
- 我现在是不是只是“先收着”，还没有整理出结论？

如果是，就写成素材型，并且标 `raw`。

### 3. 统一 frontmatter 规则

Allen 当前主库已经有比较稳定的 frontmatter 习惯，后续继续沿用：

```yaml
---
title:
source:
author:
published:
created:
description:
tags:
type:
status:
---
```

#### 推荐字段约束

- `title`：与文件标题保持一致
- `source`：原始来源；若为自建笔记则写相对路径
- `author`：默认 `Allen`；如为助手整理可写 `笨笨` 或 `Allen`
- `published`：外部文章/视频/课程的发布日期；无则留空
- `created`：当前笔记建立日期
- `description`：一句话说明这篇笔记干什么
- `tags`：少量稳定标签，优先英文小写
- `type`：见下方类型建议
- `status`：见下方状态建议

#### type 建议

- `index`：索引页 / MOC
- `note`：通用知识笔记
- `learning-note`：课程/学习类沉淀
- `project`：项目推进
- `workflow`：流程规范、SOP、工作流
- `resource`：素材、摘录、参考资料
- `archive`：归档

#### status 建议

- `raw`：刚收集，未处理
- `draft`：已开始整理，但没成型
- `processed`：结构完整，可正常复用
- `archived`：历史保留，不再维护

### 4. 正文结构规范

#### 索引型模板

```markdown
# 标题

## 摘要
一句话说明这个主题覆盖什么。

## 核心内容
### 先看这些
- [[笔记A]]
- [[笔记B]]

### 子主题
- [[笔记C]]
- [[笔记D]]

### 我的阅读顺序
1. 
2. 
3. 

## 可执行动作
- [ ] 补哪些入口
- [ ] 哪些旧笔记要并入

## 相关链接
- [[相关主题1]]
- [[相关主题2]]
```

#### 知识型模板

```markdown
# 标题

## 摘要
一句话结论。

## 核心内容
### 它是什么

### 它解决什么问题

### 我的理解

### 例子 / 对比

## 可执行动作
- [ ] 是否拆出子问题
- [ ] 是否补案例

## 相关链接
- [[上位主题]]
- [[相关概念]]
```

#### 项目型模板

```markdown
# 标题

## 摘要
当前项目要解决什么问题。

## 核心内容
### 目标

### 当前进展
- 已完成：
- 进行中：
- 卡点：

### 决策与约束

## 可执行动作
- [ ] 下一步 1
- [ ] 下一步 2

## 相关链接
- [[相关知识笔记]]
- [[相关会议纪要]]
```

#### 素材型模板

```markdown
# 标题

## 摘要
这是一份待处理素材，暂不代表最终结论。

## 核心内容
### 来源

### 摘录

### 为什么值得留

## 可执行动作
- [ ] 是否转成知识笔记
- [ ] 是否并入某个索引页
- [ ] 是否归档

## 相关链接
- [[相关主题]]
```

### 5. Allen 当前主库的实际落地规则

#### Rule 1：新笔记先分类，再命名

后续新增笔记前，先回答：
1. 这是索引、知识、项目，还是素材？
2. 它属于哪个主题域？
3. 是应该进入现有主题树，还是只放 Inbox？

不要再直接创建 `AI杂记`、`随手记`、`一些想法` 这种标题。

#### Rule 2：正式知识笔记一篇只回答一个问题

避免把多个问题混在一篇里，尤其不要把：
- 资料摘录
- 自己理解
- 项目推进
- TODO 清单

全部堆在同一页。

#### Rule 3：每个稳定主题必须有入口页

以下主题建议长期维持 MOC：
- `LLM 总览`
- `RAG 总览`
- `Agent 总览`
- `AI 工程学习路线`
- `自动化工作台`
- `投资研究地图`

#### Rule 4：OneNote 恢复内容默认先视作资源，不直接当正式知识

`ai-related/OneNote恢复/` 这类内容默认优先归类为：
- 待清洗资源
- 历史导入内容
- 候选知识来源

只有当内容被重写、压缩、结构化之后，才升级成正式知识笔记。

#### Rule 5：每篇 processed 笔记至少保留 2-5 个相关链接

链接的目标不是为了图谱好看，而是为了后续回看和扩展。

### 6. 后续整理优先级

#### 第一阶段：止血

先避免继续变乱：
- 新增 `00 Inbox/`
- 新增 `03 Resources/`
- 新增 `99 Archive/`
- 所有暂时不确定归属的内容先放进去

#### 第二阶段：建入口

优先补齐几个高价值 MOC：
- `AI 总览`
- `投资研究地图`
- `自动化工作台`

#### 第三阶段：清洗历史内容

重点处理：
- `ai-related/OneNote恢复/`
- 网页剪藏残留文件
- 无 frontmatter 的旧笔记
- 标题模糊的杂项笔记

### 7. 默认执行策略

以后只要笨笨帮 Allen 写 Obsidian 笔记，默认执行顺序如下：

1. 先判断笔记类型
2. 再判断归属目录
3. 套用统一 frontmatter
4. 使用“摘要 -> 核心内容 -> 可执行动作 -> 相关链接”结构
5. 若是 raw 内容，明确标 `status: raw`
6. 若是稳定主题，优先接入现有 MOC 或补一个 MOC

## 可执行动作

- [ ] 新建 `00 Inbox/` 目录，作为临时缓冲区
- [ ] 新建 `03 Resources/` 目录，集中放未内化资料
- [ ] 新建 `99 Archive/` 目录，承接历史导入和旧资料
- [ ] 后续逐步把 `OneNote恢复/` 中值得保留的内容重写为正式知识笔记
- [ ] 后续新增 Obsidian 笔记时，优先调用与本方案对应的 skill
