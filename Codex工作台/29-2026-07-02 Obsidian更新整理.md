# 2026-07-02 Obsidian更新整理

## 摘要

今天的 Obsidian 更新集中在 **Codex / Agent 工作台治理** 这一条线上，主要做了两类事：

1. **补齐运行态认知**：把最近新增进入运行态的 operator 和“迭代再开发 → 流水线复用”的新规则，补进已有总览与计划笔记。
2. **新增方法论研究**：新建一篇 `OpenSpec 证据链体系研究`，拆解同事那套让 Agent 不再懒散的对象化工作流、证据链闭环和与本地四层框架的结合方式。

如果只看今天最值得记住的一句话，就是：

> 本地四层 Agent 框架负责“怎么跑流程”，OpenSpec 证据链负责“结论怎么落固定槽位、怎么挂证据、怎么被门禁校验”；两者拼起来，才是更稳的长期工作流。

### 2. 今天新增补齐的运行态能力

今天补进存量笔记的 active operator 共有 3 个：

- `tool_cls_log_query_operator`
  - 用于国内腾讯云 CLS 日志查询
  - 支持 trace / message / CLS 链接输入
  - 输出重点是按 `service -> manager -> center` 链路提取第一异常点
- `tool_intl_pof_sh_confirm_operator`
  - 用于海外 POF 收货单满数量确认前的导出核查、curl 组装、校验 SQL 与风险拦截
- `tool_bk_pipeline_operator`
  - 用于蓝鲸需求流水线与服务子流水线操作
  - 今天最关键的新规则也围绕它：**继续开发默认不新建主流水线，而是复用原需求流水线**

### 3. 今天最重要的新规则：迭代再开发的流水线复用口径

今天对“迭代再开发”这条线，沉淀出了更明确的默认规则：

1. 继续开发默认**不新建** BK 主流水线；
2. `push` 后默认**复用原需求流水线**；
3. 只重跑**本次实际改动影响的已有服务子流水线**；
4. 如果改到 **POM / `hsp-invoice` 这类 Maven 共享依赖**，要先 `deploy/publish` 依赖包，再重跑原需求下所有已有服务子流水线；
5. 默认只确认目标子流水线进入 `RUNNING` 且没有即时任务报错，不默认等待全量终态；
6. 原提测文档只在服务、设计、接口、前后端/外部对接、测试范围、风险说明或验证证据变化时更新。

这条规则的重要性在于：它把“继续开发时到底怎么处理流水线”从经验口径，升级成了明确可复用的工作流规则。

### 4. 今天新增研究：OpenSpec 证据链到底值得抄什么

[[28-OpenSpec证据链体系研究]] 里最值得长期复用的，不是某个具体目录，而是这 4 条纪律：

1. **能查的不记，只记查不到的**
   - 减少长 prompt
   - 强迫 agent 自查仓库 / 表结构 / 命令
2. **一切工作先挂对象**
   - `initiative / investigation / plugin-project`
   - 先有对象，再有文档、代码和证据
3. **固定文档槽位，禁止平行 md 树**
   - 结论进固定文件
   - 进度进固定文件
   - 过程产物进 `artifacts/`
4. **每句结论挂证据链**
   - 事实 / 推断 / 待验 / 决策分级
   - claims 台账可被脚本校验、可被新证据推翻

今天这篇研究最核心的迁移结论是：

> 本地四层框架补的是“运行引擎”，OpenSpec 补的是“证据纪律”；后续最值得做的不是二选一，而是把证据链当作四层框架的产物规范。
