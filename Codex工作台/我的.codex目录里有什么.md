# 我的.codex目录里有什么

## 摘要

`~/.codex` 根目录现在的问题不是文件太多，而是运行时文件、配置文件、备份文件、缓存文件、能力目录混在一起，肉眼看会觉得“很多奇怪后缀”。真正安全的整理方式不是直接删，而是先分层：

1. 明确保留的运行核心；
2. 可以归档但不建议直接删的备份；
3. 可以重点清查的缓存和临时目录；
4. 明确不要碰的 SQLite 伴生文件。

本笔记先解释各类文件/目录的作用，再给出一份可以直接执行的整理清单。

## 核心内容

### 当前 `~/.codex` 根目录概览

根目录当前可见条目包括：

- 配置类：`config.toml`、`config.toml.bak.*`、`AGENTS.md`
- 状态类：`.codex-global-state.json`、`history.jsonl`、`session_index.jsonl`
- SQLite 类：`goals_1.sqlite`、`logs_2.sqlite`、`state_5.sqlite` 及其 `-wal` / `-shm`
- 目录类：`skills`、`superpowers`、`hooks`、`memories`、`vendor_imports`
- 缓存/临时类：`.tmp`、`tmp`、`cache`、`log`、`shell_snapshots`
- 备份类：`skills_backup`、`skill-duplicates-archive`

### 各种“奇怪文件/后缀”的作用

#### 1. `config.toml` 与 `config.toml.bak.*`

- `config.toml`
  当前主配置，正在生效。
- `config.toml.bak.20260317`
- `config.toml.bak.20260327-autostart`
- `config.toml.bak.20260408-155242-ecc-all`
- `config.toml.bak.20260419-153804`

这些 `.bak.*` 文件都是历史配置备份快照，用来回滚到某个时间点或某种实验状态。它们不是垃圾，但散在根目录会增加辨识成本。

#### 2. `*.sqlite` / `*.sqlite-wal` / `*.sqlite-shm`

代表文件：

- `goals_1.sqlite`
- `logs_2.sqlite`
- `state_5.sqlite`
- 对应的 `-wal`
- 对应的 `-shm`

解释：

- `*.sqlite` 是主数据库文件
- `*.sqlite-wal` 是 SQLite 的预写日志
- `*.sqlite-shm` 是 SQLite 的共享内存辅助文件

这类文件不是脏文件，也不是单纯缓存。只要 Codex 还在运行，就不应该手工删除，尤其是 `-wal` 和 `-shm`。

#### 3. `*.json` / `*.jsonl`

代表文件：

- `.codex-global-state.json`
- `.codex-global-state.json.bak`
- `history.jsonl`
- `session_index.jsonl`
- `version.json`
- `auth.json`

解释：

- `.json` 通常是状态文件或配置状态
- `.jsonl` 通常是追加式流水记录
- `.bak` 是备份

其中：

- `history.jsonl` 是会话历史流水
- `session_index.jsonl` 是会话索引
- `auth.json` 是认证状态
- `.codex-global-state.json` 是全局状态

这类文件不要因为“看起来像文本”就轻易删。

#### 4. 临时/缓存/运行目录

代表目录：

- `.tmp`
- `tmp`
- `cache`
- `log`
- `browser`
- `node_repl`
- `shell_snapshots`
- `sessions`
- `sqlite`

这类目录通常和：

- 运行中间产物
- 调试快照
- 浏览器/REPL 状态
- 缓存
- 临时文件

有关。它们是最值得做二次清查的对象，但也最不适合“盲删”。

#### 5. 能力与系统目录

代表目录：

- `skills`
- `superpowers`
- `hooks`
- `plugins`
- `rules`
- `memories`
- `vendor_imports`
- `commands`

这些基本都属于系统能力层，不是目录垃圾。

其中：

- `skills` 是用户级核心能力库
- `superpowers` 是流程增强系统
- `hooks` 是自动化钩子
- `memories` 是长期记忆
- `vendor_imports` 是导入能力

这类目录默认都应该保留。

#### 6. 备份/归档目录

代表目录：

- `skills_backup`
- `skill-duplicates-archive`

解释：

- `skills_backup` 是技能备份树，不是简单缓存
- `skill-duplicates-archive` 是重复技能归档区

它们不一定需要长期放在根目录，但也不该直接清空。

### 关于 `config.template.toml` 的历史角色

这个文件不是系统自动生成的缓存，也不是运行时强依赖。

它当时是个人多 agent 骨架的模板配置，作用是：

- 提供多 agent registry 参考
- 与 `~/.codex/agents/` 目录中的其他骨架文档配套
- 在主 `config.toml` 合并前做设计和审阅

但按当前真实状态：

- 这个过渡文件已经移除
- 当前唯一运行态配置真源是 `~/.codex/config.toml`

所以你现在再看 `~/.codex` 时，不应该继续把 `config.template.toml` 当成现存文件或维护对象；它只是一段历史阶段。

### `~/.codex` 目录整理清单

#### A. 明确保留，不动

- `config.toml`
- `AGENTS.md`
- `agents`
- `skills`
- `superpowers`
- `hooks`
- `plugins`
- `rules`
- `memories`
- `vendor_imports`
- `commands`
- `auth.json`
- `history.jsonl`
- `session_index.jsonl`
- `.codex-global-state.json`
- 所有 `*.sqlite`
- 所有 `*.sqlite-wal`
- 所有 `*.sqlite-shm`

#### B. 可归档整理，但不建议直接删

- `.codex-global-state.json.bak`
- `config.toml.bak.20260317`
- `config.toml.bak.20260327-autostart`
- `config.toml.bak.20260408-155242-ecc-all`
- `config.toml.bak.20260419-153804`
- `skills_backup`
- `skill-duplicates-archive`

建议动作：

- 后续统一迁移到 `backups/` 或 `archive/` 目录
- 保留时间戳命名
- 不要直接在根目录长期平铺

#### C. 可以重点清查的目录

- `.tmp`
- `tmp`
- `cache`
- `log`
- `shell_snapshots`
- `sessions`
- `browser`
- `node_repl`

建议动作：

- 先检查是否有当前进程持有
- 再决定是清空、保留最近 N 天，还是迁移归档

#### D. 明确不要碰的对象

- `goals_1.sqlite-shm`
- `goals_1.sqlite-wal`
- `logs_2.sqlite-shm`
- `logs_2.sqlite-wal`
- `state_5.sqlite-shm`
- `state_5.sqlite-wal`

原因：

- 它们是 SQLite 正常运行的一部分
- 手工删除容易破坏运行中状态或造成数据不一致

### 当前建议的整理顺序

1. 先建立一份 `保留 / 可归档 / 可清查 / 不要动` 清单。
2. 再处理 `config.toml.bak.*` 和 `.json.bak` 这类备份。
3. 最后清查 `.tmp`、`tmp`、`cache`、`shell_snapshots`。
4. 暂不处理任何 `*.sqlite*` 文件。

## 可执行动作

1. 保持当前 `config.toml`、`skills`、`superpowers`、SQLite 文件不动。
2. 下一轮优先整理备份文件：
   - `config.toml.bak.*`
   - `.codex-global-state.json.bak`
3. 再单独做一次缓存目录排查：
   - `.tmp`
   - `tmp`
   - `cache`
   - `shell_snapshots`
4. 若要真正做根目录瘦身，优先通过“新建 `backups/` 归档目录”来收拢，而不是直接删除。
