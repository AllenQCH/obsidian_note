### 10. 新的开发测试门禁

当前 multi-agent 运行态新增了一条开发收口规则：

> 生产代码开发任务，不能只靠编译或单测通过就宣布结束。

现在默认要求两层验证都过：

1. 代码级验证

   - 仓库基线中的上下文测试通过，或明确记录为环境债务

   - 只建最小表

- 数据落表
- 定时任务

### 11. 通用本地闭环测试规范

当前又补齐了一条更具体的测试 agent 规则：

> 当需求涉及真实业务数据路径时，测试 agent 不能只生成单测或只验证服务启动；必须从真实入口触发，尽量跑真实内部链路，只 mock 外部依赖，并用本地业务结果证明改动生效。

适用场景包括：

- 审核、审批、幂等、补偿流程
- 管理页查询、编辑、导出

本地闭环辅助代码允许入仓，但必须满足：

- 集中在 `config/localtest` 或等价本地测试目录
- 只在 `local-*` profile 或显式开关下生效
- 不影响默认 profile、测试环境、预发环境和生产环境
- OpenSpec 或需求记录必须写清楚本地资产、启用条件、执行命令、验证证据和环境边界
- 如果满足以上条件，并且该资产已经真实用于闭环验证，则默认纳入提交候选；后续统一按“正式需求代码 + 测试 + OpenSpec + 已隔离本地闭环资产”的方案处理

### 12. 新增的多仓库开发执行顺序

对多仓库开发需求，当前又补齐了一条更实用的执行顺序：

1. 先完成路由、OpenSpec 和文件级 diff plan。
2. 迭代开发默认不再逐仓库确认来源分支和目标需求分支名。
3. 对每个实际改动仓库先 `fetch origin`，切回 `master`，`pull --ff-only origin master`。
4. 再基于最新 `master`，按既定命名规则创建规范需求分支。
5. 分支创建完成后，先在每个实际改动仓库里补服务级 OpenSpec，再开始编码实现。

只有用户指定非 `master` 来源分支、分支名无法按需求编码/需求名生成、涉及仓库不明确、远端同名分支存在新建/复用歧义，或本地/远端分支状态冲突时，才回到人工确认。

### 13. 分层映射边界规则

当前又固化了一条更具体的多层后端分工规则：

> 当问题只是展示字段重命名、DTO 别名兼容或前端响应整形时，不要默认把改动下推到 `center` 或 mapper 层；优先在 `manager`、`backend`、`convert` 这类上层编排/展示层处理。

默认下推到 `center` 的前提应当是：

- SQL 过滤条件缺失
- 返回列缺失
- 持久化契约错误
- 下层领域对象本身不完整

如果只是 `salesInfo -> sellerInfo` 这类展示属性改名，则默认不在 `center` mapper 层改。

这条顺序适用于多服务的 `新项目开发` 和 `迭代开发`，可以减少“分支基线不一致”以及“OpenSpec 只留在组级目录、没落到代码仓库”的问题。

### 14. 人工控制简化规则

当前 multi-agent 流程已经把人工审核点从“每一步都确认”收敛为“按风险暂停”：

- 自动推进：只读分析、本地文件读取、本地 OpenSpec / 需求记录更新、确定性路由摘要、迭代开发默认从最新 `master` 创建规范需求分支、本地实现与验证、需求/设计确认且测试通过后的 commit/push、用户明确要求提测后的 BK 需求流水线创建与在线提测文档创建。
- 一次性授权：迭代开发默认把路由、需求/设计方向、允许的外部写动作合并成前置人工控制区；确认后，不再把拉 `master`、创建规范分支、写代码、写测试、跑测试、测试通过后的 commit/push 拆成重复人工确认。默认路由交付是输出单行启动命令和完整 prompt，由用户自行开新窗口粘贴，不作为额外人工确认点。
- 测试失败处理：agent 先自动定位失败原因；只有修复方案会改变已确认设计、扩大范围、改变接口契约，或需要接受失败继续推进时，才回到人工确认。
- 必须暂停：路由或工作区歧义、来源/目标分支缺失、破坏性 Git、强推、删除分支、合并主干、审批放行、环境晋级、停止/重试生产部署、在线文档删除/移动、同名流水线新建还是复用不明确。

### 15. 迭代再开发自主闭环规则

`existing_feature_continuation` 的重点是前期沟通，而不是重新跑完整开发交付。

默认流程：

1. 路由确认为 `existing_feature_continuation`。
2. 读取已有 OpenSpec、分支、历史提交、历史测试、提测文档和流水线记录。
3. 人工确认本次续改目标，以及是否改变原设计。
4. 确认后自动修改代码、生成或更新测试规则、执行代码级验证和必要的本地闭环验证。
5. 测试通过后自动 commit/push。
6. 默认不新建 BK 流水线；如果新增服务、仓库、部署单元或服务分支条目，优先把新增服务补进原有需求流水线。
7. 只有提测文档中的涉及服务、设计、接口、前端对接、外部对接、测试范围、风险说明或验证证据发生变化时，才更新原提测文档。

需要人工确认的情况：

- 不清楚复用哪个 OpenSpec、分支、提测文档或流水线记录。
- 本次续改目标可能改变原设计。
- 需要新建分支而不是复用旧分支。
- 原流水线无法定位，或新增服务的名称、分支、项目、模板、部署语义不明确。
- 测试失败后的修复方案会改变设计、扩大范围、改变接口契约，或需要接受失败继续推进。
- 用户明确要求重新提测、创建新流水线或新 release handoff。

### 16. codex-downloads 统一 OpenSpec 归档层

当前新增了一个跨项目需求总索引：

- `/Users/heytea/Documents/HeyTea/code/codex-downloads/openspec/`

用途：

- 一眼查看最近做过的需求。
- 汇总涉及仓库、服务、分支、commit、测试、流水线、提测文档、前端对接和外部对接。
- 归档不属于单一服务仓库的交付材料。
- 给 `新项目开发`、`迭代开发`、`迭代再开发` 和提测交付提供统一导航入口。

目录规则：

- `active/`：进行中的需求索引。
- `archived/<year>/`：已完成需求索引。
- `attachments/<需求编号>/`：前端对接文档、服务对接文档、提测文档、BK pipeline JSON、AliDocs evidence JSON 等附件。
- `templates/demand-index-template.md`：统一索引模板。

边界：

- 统一 OpenSpec 不替代服务级 OpenSpec。
- 服务级 OpenSpec 仍放在实际服务仓库，随服务代码 commit/push。
- 统一 OpenSpec 只做跨项目导航、交付材料归档和最近工作总览。
- 不写入密钥、cookie、token、生产数据导出或敏感个人信息。
