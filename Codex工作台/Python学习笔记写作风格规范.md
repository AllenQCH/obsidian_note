# Python学习笔记写作风格规范

## 摘要

这份规范不是在定义“标准技术文档怎么写”，而是在沉淀我自己的 Python 学习笔记风格。

以后无论是 Codex 还是其他 agent，只要是在 Obsidian 里帮我写 Python、FastAPI、项目代码分析、语法理解类笔记，都优先遵循这套风格，而不是默认写成官方文档风、百科定义风或过度结构化的知识卡片风。

核心目标只有一个：让笔记第一次看就容易懂，第二次回看也容易快速捡起来。

## 核心内容

### 适用范围

- `ai-related/13-python学习/` 目录下的 Python 基础、FastAPI、项目分析类笔记。
- 其他目录中，只要主题本质上属于“Python 学习讲解型笔记”，也优先套用这套风格。
- 这套风格优先级高于通用技术笔记风格，但不替代 Obsidian 全局属性规范。

### 风格定位

- 不是教科书。
- 不是官方参考文档。
- 不是纯摘录。
- 更像“有人坐在旁边，把这段代码和这个概念讲明白”。

### 核心写法规则

#### 1. 开头先讲本质，不先铺背景

- 第一段优先回答：这是什么。
- 第二段优先回答：它本质上在做什么。
- 不要一开始就写发展历史、完整定义、生态全景。

推荐写法：

```text
`import` 本质上是在做两件事：
1. 找到模块
2. 把需要的名字放到当前作用域
```

不推荐写法：

```text
`import` 是 Python 语言中用于导入模块、对象、包和命名空间的重要关键机制，在软件工程中具有重要作用。
```

#### 2. 先讲结论，再讲展开

- 先给一句人能抓住的结论。
- 再按小节逐步拆开。
- 不要求一次讲完整体系，先让读者进入理解状态。

#### 3. 尽量讲人话

- 多用“意思是”“本质上”“可以理解成”“这里其实是在”。
- 少用一上来就很重的术语定义。
- 如果必须用术语，立刻补一句白话解释。

#### 4. 一段只讲一个点

- 每段尽量短。
- 一个小标题只解释一个核心动作或一个核心概念。
- 宁可分得更碎，也不要堆成长段。

#### 5. 优先用“拆解式”结构

常见结构：

- `这是什么`
- `为什么需要`
- `本质`
- `逐行解释`
- `整体运行流程`
- `最终整体理解`

这类结构比“定义 / 分类 / 特征 / 优缺点”更符合当前风格。

#### 6. 如果有代码，优先从代码反推概念

- 先贴最小代码。
- 再逐行讲。
- 再抽象出规律。

顺序优先是：

```text
代码示例
-> 每一行在干什么
-> 组合起来整体发生了什么
-> 这个模式在真实工程里意味着什么
```

#### 7. 可以做轻量类比，但不要失控

- 对 Java / Spring 背景的人，可以适度类比。
- 类比只用于降低理解门槛，不要把重点转成另一门语言教程。

适合的类比：

- `FastAPI` 类似 `Spring Boot`
- `Depends` 类似依赖注入入口
- `dict` 类似 `Map`
- `FastAPI()` 类似 `new`

#### 8. 解释语法时，也要补工程意义

- 不只说“它是什么语法”。
- 还要补一句“为什么在 FastAPI / AI / 工程里常见”。

例如写 `async` 时，不只解释异步定义，还应补：

- 数据库等待
- 网络请求
- AI 调用
- streaming

#### 9. 复杂主题要补运行流程

涉及下面这些主题时，尽量补一个流程视角：

- FastAPI 请求处理
- 路由匹配
- 依赖注入
- 中间件
- 模块调用关系
- 项目入口执行链路

推荐写法：

```text
请求进来
↓
路由匹配
↓
依赖执行
↓
函数执行
↓
返回 JSON
```

#### 10. 允许重复，但重复要有层次

- 第一遍：一句话本质
- 第二遍：拆开解释
- 第三遍：整体再收束

不是机械重复，而是帮助理解闭环。

#### 11. 结尾尽量收一个“最终理解”

- 结尾最好不是突然停住。
- 用一个短总结，把前面的点重新串起来。

常见收口方式：

- `总结`
- `最终整体理解`
- `一句话理解`

### 表达风格规则

- 语气偏讲解，不要太像规范文件。
- 保持“有人在带你理解”的感觉。
- 可以直接说“这里是重点”“这一步很关键”。
- 避免空洞形容词，比如“强大”“优雅”“灵活”单独出现而不解释。

### 格式规则

- 标题层级以 `#`、`##`、`###` 为主。
- 多用短段落。
- 多用短 bullet。
- 代码块只放当前解释需要的最小片段。
- 能一句话说清的，不强行做大表格。

### 不推荐的写法

- 一上来先写完整定义。
- 长篇背景介绍后才进入正题。
- 过度追求术语严谨，牺牲第一次阅读体验。
- 大段堆砌知识点，没有“理解顺序”。
- 写成官方文档翻译腔。
- 写成过度完整的八股笔记。

### 给 agent 的直接约束语

以后如果要让 agent 按这套风格写，可以直接附上下面这段：

```text
请按我的 Obsidian Python 学习笔记风格来写：
1. 开头先用 1-3 句话讲本质，不先铺背景。
2. 优先讲人话，多用“意思是 / 本质上 / 可以理解成”。
3. 用短段落和小标题拆解，每段只讲一个点。
4. 如果有代码，先贴最小示例，再逐行解释。
5. 复杂主题补“整体运行流程”或“执行链路”。
6. 可以适度用 Java / Spring 做轻量类比，但不要展开太深。
7. 不只解释语法，还要解释它在真实工程里为什么重要。
8. 结尾加一个“总结”或“最终整体理解”来收束。
9. 风格更像教学讲解，不像官方文档。
10. 可读性优先于完备性，宁可拆碎，也不要写成长段。
```
