# trace日志真实任务怎么记录

## 摘要

这篇只解决一个问题：

> 当你手上真的拿到一个 `traceId`，现在第一步到底做什么，第二步做什么？

这不是再讲理论，而是一张一页执行卡片。

## 执行顺序

### 第 1 步：先判断这次输入够不够开始

打开：

- [[trace日志最小执行入口]]

只看三件事：

1. 有没有 `trace_id`
2. 有没有 `source_type`
3. 有没有至少一类机器可读证据：
   - `raw_logs`
   - `platform_results`
   - `export_file_path`

如果这三件事不满足：

- 这次先不算标准化 case
- 先补输入

### 第 2 步：如果能开始，就按样例组织格式

打开：

- [[trace日志案例填写示例]]

目的不是抄内容，而是抄结构。

重点照着组织：

- 任务锚点
- 输入形状
- 已确认事实
- 决策字段
- gate 结果
- 边界检查

### 第 3 步：把这次真实任务正式记下来

打开：

- [[trace日志真实案例模板]]

把真实内容填进去。

这一步的目标是：

- 不靠回忆
- 不只写结论
- 把 facts、decision、uncertainty 分开

### 第 4 步：判断这次能不能算晋升证据

还是在模板里看最后几项：

- 这次是否计入激活证据
- 为什么
- 下次还缺什么

如果这次只是：

- 输入不完整
- 时间线拼不起来
- 根本没法判断 gate

那它就只能算上下文，不算有效晋升证据。

### 第 5 步：等攒够 3 个真实 case，再做正式评审

打开：

- [[trace日志agent晋升评审]]

这一页不要过早用。

只有当：

1. 已经有至少 3 个真实 case
2. case 格式比较稳定
3. 你真的在判断能不能进 active

才用它。

## 一句话版

如果只记最短顺序：

```text
21 看能不能开始
-> 22 看怎么填
-> 15 正式记录
-> 23 攒够 3 个 case 后再评审
```

## 不要做的事

- 不要拿到 `traceId` 就直接下根因结论
- 不要跳过 `21`，直接把任何排查请求都算标准化 case
- 不要只看 `22` 的样例，不做 `15` 的正式记录
- 不要在只有 1 个 case 时就提前用 `23` 决定晋升
