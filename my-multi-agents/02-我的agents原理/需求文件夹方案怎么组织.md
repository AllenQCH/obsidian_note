# 需求文件夹方案怎么组织

## 摘要

开发需求时 agent 产出的文件统一落在**按需求号命名的文件夹**里，但内部分成两层：**需求 -> 轮次 -> 本轮文档**。

- 位置：`/Users/heytea/Documents/myHeytea/code/codex-downloads/openspec/`
- 需求根只放 `index.md`（总览+轮次台账+当前轮次指针）和 `requirement.md`（需求本身，跨轮稳定）。
- **每一轮开发 = 一个 `iterations/<序号>-<role>-<简述>/` 子文件夹**，迭代 role 和迭代再开发 role 各成一轮，本轮的方案/进度/测试/交付/证据都在里面。
- 开发中在 `active/`；开发完整个需求文件夹**整体搬到** `archived/<年>/`。

这解决了之前的痛点：多轮开发时文件重复、命名不统一、找不到该操作哪个——现在每轮物理隔离，`index.md` 指针指向当前轮。

## 三个核心问题的答案

- **开发中 vs 开发完的区别**：开发中在 `active/`，`index.md` 状态 developing、有「当前轮次指针」和未勾选 tasks；开发完整体移到 `archived/<年>/`，状态 done、冻结只读。
- **每次开发要不要分文件夹**：要。每轮一个编号子文件夹，轮内文件名固定（design/tasks/verification/delivery），靠轮次号定位，不再重复混乱。
- **测试两种怎么放**：每轮 `test/` 下分 `local-chain/`（本地起服务跑链路）和 `test-env/`（测试环境可执行 SQL/curl 包 + guide）。

## 文件夹结构

```text
openspec/
  README.md                       # 约定入口（真相源）
  rules/default-dev-rules.md      # 跨需求默认开发规则（agent 开工先读，人维护）
  templates/
    demand-folder/                # 需求脚手架：index.md + requirement.md + iterations/
    iteration/                    # 单轮脚手架：新一轮照抄这个
  active/
    <需求号>-<需求名>/
      index.md          # 总览+轮次台账+当前轮次指针（现在该动哪个）
      requirement.md    # 需求本身（跨轮稳定）
      iterations/
        01-iterative-<简述>/       # 第1轮（迭代 role）
          design.md tasks.md verification.md delivery.md
          test/
            local-chain/plan.md    # 测试A：本地起服务跑链路
            test-env/              # 测试B：测试环境可执行包
              guide.md  sql/  curl/
          evidence/                # 本轮证据/提测/对接文档
        02-continuation-<简述>/    # 第2轮（迭代再开发 role），结构同上
  archived/<年>/<需求号>-<需求名>/   # 收口后整体搬来
```

服务级 OpenSpec 仍在各服务仓 `openspec/` 随代码走，`index.md` 回链。

## 每一轮怎么开

- 新需求第一轮：拷 `templates/demand-folder/` 建需求 → 拷 `templates/iteration/` 到 `iterations/01-iterative-<简述>/`。
- 继续开发/再开发/中途 bug：**不新建需求文件夹**，按需求号找到需求，读 `index.md` + 上一轮 `design.md`，**新开一轮** `iterations/<NN>-continuation-<简述>/`，更新 `index.md` 台账与指针。
- 不改已关闭轮次的文件。

## 测试两种（每轮 test/ 下）

| 类型 | 位置 | 内容 | 谁执行 |
|---|---|---|---|
| A 本地链路 | `test/local-chain/` | plan.md（起哪些服务、mock 什么、验证点）+ 结果 | test role 本地跑 |
| B 测试环境 | `test/test-env/` | guide.md（顺序指引）+ sql/ + curl/，按测试范围整理成可执行包 | 人照 guide / 未来「测试环境 role」 |

`verification.md`（轮次根）= 两种测试的总结与 pass/fail 裁决。

## 其它文档放哪

| 文档类型 | 放哪 |
|---|---|
| 需求级、跨服务、给人看(提测文档、测试报告、对接、BK 配置、AliDocs 证据) | 本轮 `evidence/handoff/`(提测/对接)与 `evidence/test/`(测试报告)，`verification.md`/`delivery.md` 做索引 |
| 本轮测试环境可执行包(整理版 SQL/curl) | 本轮 `test/test-env/` |
| 仓级原始 fixture(随代码走) | 该服务仓 `docs/test-data/`，在 `verification.md` 链接，不复制 |

## 三层「规则」各归各位

| 内容 | 放哪 |
|---|---|
| 跨需求默认规则(门禁、JDK、依赖包前缀、红线) | `openspec/rules/default-dev-rules.md` |
| 项目级持久规则(某仓是不是依赖包等) | 各服务仓 `AGENTS.md` |
| 本轮技术决策/方案 | 该轮 `design.md` |

## agent 怎么用（已写进框架）

- 路由 `control_request_router`：解析 `demand_folder` + `iteration_binding`（新轮次），提示先读默认规则。
- 规划 `stage_task_planner`：新开发建需求+第一轮；已有需求读历史再开新一轮，写本轮 design/tasks。
- 排查 `stage_investigation_planner`：按需求号读历史，bug 修复作为新一轮 continuation。
- 执行 `stage_execution_runner`：更新当前轮 tasks、证据落当前轮 evidence/。
- 测试 `stage_test_runner`：产出 local-chain + test-env 两种物料，写当前轮 verification.md 裁决。
- 收口 `stage_closeout_reporter`：填当前轮 delivery.md、更新 index 台账；整个需求收口才整体移 archived。

## 关键点

- **稳定 key = 需求号**：一个需求号一个文件夹，continuation 靠它找历史。
- **轮次隔离**：多轮开发各自一个 iterations 子文件夹，不重复、不乱名，指针告诉你动哪个。
- **测试两分**：本地链路 + 测试环境可执行包，分开放，为将来「测试环境 role」留好入口。
