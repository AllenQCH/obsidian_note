# 本机资源注册表怎么查

## 摘要

现在本机的 Agent、Skill、MCP 和 Plugin 由统一构建器扫描，HTML 和 Markdown 从同一份规范化注册表生成。

运行事实来源：

- Agent / MCP / Plugin：`~/.codex/config.toml` 和 Agent TOML
- Skill：各 Skill 根目录中的 `SKILL.md`
- 人类职责说明：`~/.codex/agents/resource_metadata.json`

统一查询数据：

- `~/.codex/agent-catalog-runtime/resource_registry.json`

查询页：

- `file:///Users/heytea/.codex/agent-catalog-runtime/agent_catalog.html`

Markdown 视图：

- `~/.codex/agents/docs/agent-registry.md`

默认 Obsidian 地址根：

- `my-multi-agents/02-我的agents原理/本机资源怎么注册和查询/`

当前配置：

- `tasks/20260630-agent-catalog/agent_catalog.config.json`
- `vault_note_root = my-multi-agents/02-我的agents原理/本机资源怎么注册和查询`

2026-07-18 最近一次验证时，页面收录 **293 条资源**：

- `skill`: 250
- `agent`: 9
- `tool`: 19
- `mcp`: 6
- `plugin`: 9

其中 `~/.hermes/skills` 递归发现 **153 个 Hermes skills**。这类资源会标记为：

- `runtime = hermes`
- tag 包含 `hermes-skill`

页面搜索 `hermes` 或 `hermes-skill` 即可只看 Hermes 能力。

注意：HTML 页面上的筛选 tab 是使用口径，不是严格 `kind` 口径。比如 `agent` tab 会把 configured tool agents 一起展示，所以 tab 数量会和这里的真实类型数量略有差异。

## 核心内容

### 当前登记范围

- `skill`
- `agent`
- `tool`
- `mcp`
- `plugin`

Skill 默认递归扫描：

- `~/.codex/skills`
- `~/.agents/skills`
- `~/.claude/skills`
- `~/.hermes/skills`
- `~/.codex/plugins/**/skills`

### 当前字段

每条资源至少记录：

- `name`
- `desc`
- `source`
- `obsidian`
- `tags`
- `scope`
- `extra`
- `status`
- `source_hash`
- `details.responsibility`
- `details.next_agents`
- `details.must_not`

### 使用方式

全量同步并同时重建 registry、HTML、JSON 和 Markdown：

```bash
/Users/heytea/.codex/agent-catalog-runtime/sync_agent_catalog.sh
```

只检查运行源、统一注册表和生成物是否一致：

```bash
/Users/heytea/.codex/agent-catalog-runtime/check_agent_catalog.sh
```

直接打开查询页：

```bash
/Users/heytea/Documents/myHeytea/codex-workspace/tasks/20260630-agent-catalog/open_agent_catalog.sh
```

### 设计取舍

- 页面只做查询，不做重前端编辑器。
- HTML 和 Markdown 都是只读生成视图，不允许反向覆盖运行配置。
- `resource_registry.json` 是两个视图共同读取的规范化查询数据，不替代 Codex 运行配置。
- 自动扫描负责 Agent、Skill、MCP、Plugin 的新增、修改和删除。
- `resource_metadata.json` 负责职责、下游 Agent 和禁止事项等人类语义。
- 自动任务监听常用运行源路径，并每 5 分钟兜底同步。
- `check` 会发现运行源漂移、孤立元数据、缺失 Agent 文件和过期视图。

### Hermes 内部网页任务的限时 Skill 发现规则

这套探索规则现在由 Hermes 的 `opencli-browser-webapp-exploration` Skill 持有，不放在 Codex multi-agent 路由中。公司内部网页经常是开源产品的二次改版；遇到没有本地专用 Skill 的网页操作时，Hermes 默认顺序是：

```text
本地资源注册表 / tool-agent-matrix
-> 已安装 Codex / Hermes Skill
-> 限时 find-skills
-> 浏览器 / opencli 硬处理
```

只有同时满足以下条件才搜索外部 Skill：

- 本地没有合适能力；
- 页面或流程像可复用的开源产品类别；
- 操作是重复、多步骤，使用专用 Skill 能明显降低成本。

时间和范围限制：

- 本地查询不超过 15 秒；
- `find-skills` 最多 2 次查询、总计不超过 90 秒；
- 最多审查 3 个候选，出现第一个可信匹配就停止；
- 简单一次性查看/点击、预计 2 分钟内能完成、紧急排障或已有本地能力时不触发搜索；
- 搜不到或超时就回退浏览器/opencli，不让搜索阻塞主任务。

安全边界：外部搜索只能使用通用产品或功能关键词，不能携带内部 URL、公司/租户名称、ID、截图、表结构、Cookie、Token 或业务数据。搜索到但尚未安装的第三方 Skill 只是候选，安装前仍需人工确认。

实现位置：

- `~/.hermes/skills/software-development/opencli-browser-webapp-exploration/SKILL.md`
- 底层发现能力：`~/.hermes/skills/find-skills/SKILL.md`

## 可执行动作

- 以后新增、修改或删除 `skill / agent / tool / mcp / plugin` 时，只修改真实运行源；需要立即刷新时运行统一 `sync_agent_catalog.sh`。
- 如果职责、调用关系或安全边界变化，同时修改 `~/.codex/agents/resource_metadata.json`。
- 完成后运行 `check_agent_catalog.sh`，输出 `catalog check passed` 才算三套视图一致。
- 如果要把某个资源真正沉淀成独立 Obsidian 笔记，就沿用 `my-multi-agents/02-我的agents原理/本机资源怎么注册和查询/<kind>/<name>.md`。
