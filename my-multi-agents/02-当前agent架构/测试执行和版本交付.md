# 测试执行和版本交付

## 摘要

本地测试、版本交付和测试环境验证是三个不同责任面；每一步都必须留下实际目标、命令、退出状态和观察结果。

## 本地测试

`stage_test_runner` 根据用例启动项目登记的本地依赖和服务，必要时使用 Docker；通过真实 Controller、Consumer、Job 或 Runner 触发链路，并验证 API、DB、消息、文件或日志结果。仅启动容器或端口不算通过。

## 版本交付

`stage_version_delivery` 在一个阶段内完成：解析受影响服务和依赖包、准备权限 SQL 与提测材料、执行已授权的依赖发布和 BK 流水线动作、记录 Commit 和交付证据。它不再拆成 Delivery Orchestrator、Release Observer 或 Closeout Reporter。

AliDocs 六 Sheet 在线提测文档由 `stage_version_delivery` 按需委托 `tool_alidocs_test_delivery_agent`。该 Tool Agent 只负责参数校验、dry-run、创建计时、Sheet/单元格回读和结构化证据；不负责 Commit、依赖发布、SQL 或 BK 流水线。底层 `alidocs-test-delivery-doc` Skill 和 `opencli heytea-alidocs` 继续作为直接能力与降级入口。

## 测试环境

`stage_test_environment_runner` 先确认测试环境部署的精确版本，再检查现有数据；缺数据时生成 SQL 并按授权提工单，然后进入目标 Pod 执行 curl，最后核对 DB、日志和业务结果。失败时输出 `bug-list.md`，成功时进入 `gate_archive_ready`。
