# 02-05-RAG-Rerank与Context构建

## 摘要

- Rerank 是在初召回之后，对候选 chunk 做更精细的相关性排序。
- Context 构建是在 token 预算内，把最有价值的证据、来源和指令放进模型上下文。
- 召回解决“有没有找出来”，Rerank 解决“谁更相关”，Context 构建解决“该给模型看什么”。
- 这一步的核心风险是上下文污染：错误或无关资料进入上下文后，模型会基于错误证据回答。

## 核心内容

### Rerank 为什么重要

向量召回通常只做粗排，容易出现：

- 语义看似相近但业务上无关。
- 只命中片段中的个别词。
- 多个候选 chunk 都相似，但真正能回答问题的只有少数几个。
- chunk 太长导致部分内容相关、部分内容无关。

Reranker 通常会重新评估：

```text
用户问题 + 候选 chunk -> 相关性分数
```

然后按分数重新排序，保留更有用的资料。

### Context 构建要放什么

| 内容 | 原则 |
|---|---|
| 用户问题 | 保留原始意图，不被检索内容覆盖 |
| 系统指令 | 约束模型必须基于资料回答 |
| 证据 chunk | 只放最相关、可引用、不过期、权限允许的内容 |
| 来源信息 | 保留标题、路径、链接、更新时间，方便引用 |
| 拒答规则 | 没有资料时说明无法确认，不要编 |

### 过滤层

常见过滤顺序：

1. 权限过滤：用户不能看的资料不能进入候选集。
2. Metadata filter：业务域、系统、版本、时间、文档类型。
3. 相似度阈值：过滤明显无关的结果。
4. Rerank 阈值：只保留精排后相关的结果。
5. Token budget：按预算截断或摘要。

### 常见陷阱

- 只按相似度排序，不看权限、时间和来源。
- 把过多 chunk 塞进上下文，导致重点信息被稀释。
- 缺少引用来源，答案无法验证。
- 正确证据在上下文中间偏后，被长上下文削弱。
- 检索结果互相矛盾，但没有让模型识别冲突和说明不确定性。

## 可执行动作

- [ ] 给 RAG prompt 增加“只基于证据回答，证据不足则说明不足”的约束。
- [ ] Context 中保留每个 chunk 的 title、source、updated_at。
- [ ] 设计“无召回 / 低置信度 / 证据冲突”的回答策略。
