# 02-04-RAG-Retrieval召回与混合检索

## 摘要

- 召回的目标是把“可能相关”的 chunk 从知识库中找出来。
- 召回阶段更关注覆盖率：正确资料必须先进入候选集，后面才有重排和生成的机会。
- 只用向量召回容易漏掉精确关键词；只用关键词检索又容易漏掉同义表达。
- 企业 RAG 更常用混合检索：向量检索 + 关键词检索 + metadata filter。

## 核心内容

### 召回的位置

```text
用户问题
  -> 查询改写 / 扩展
  -> Query Embedding
  -> 初召回 TopK
  -> 过滤与合并
  -> Rerank
```

召回不是最终答案排序，它只是先拿到候选资料。

### 常见检索方式

| 方式 | 擅长 | 局限 |
|---|---|---|
| 向量检索 | 同义表达、语义相似、模糊查询 | 精确字段、编号、版本号不稳定 |
| 关键词检索 | 精确词、字段名、错误码、文档标题 | 同义表达和改写能力弱 |
| Metadata filter | 权限、业务域、时间、来源、版本过滤 | 依赖索引时写入完整元数据 |
| 混合检索 | 同时提高语义覆盖和精确命中 | 需要合并、去重和权重调参 |

### TopK 和阈值

TopK 是召回多少个候选 chunk。它不是越大越好：

- TopK 太小：正确资料可能没被召回。
- TopK 太大：噪声增多，Rerank 和 Context 成本上升。
- 相似度阈值太高：容易漏召回。
- 相似度阈值太低：无关资料进入候选集。

经验上，初召回可以相对宽松，再交给 [[02-05-RAG-Rerank与Context构建|Rerank]] 精排。

### 查询改写

查询改写用于提高召回率：

- 补充同义词。
- 把口语问题改成文档里更可能出现的表达。
- 将复杂问题拆成多个子问题。
- 补充业务域、时间、系统名等过滤条件。

查询改写要保守，不能把用户问题改成另一个问题。

### 召回失败的典型原因

- 文档没有进入知识库。
- 文档进入了，但解析或切块质量差。
- chunk 缺少标题、路径、业务域等上下文。
- 用户问法和文档表述差异太大。
- 只用了向量检索，漏掉精确关键词。
- 权限或 metadata 过滤条件配置错误。
