# 02-06-RAG-企业级落地

## 摘要

- 企业级 RAG 的难点不只是“接一个向量库”，而是知识治理、权限、更新、评估和可追溯。
- 资料质量、metadata、权限过滤和评估集通常比模型选择更影响最终效果。
- 企业 RAG 必须能回答“答案来自哪里、是否有权限、资料是否过期、为什么没答出来”。

## 核心内容

### 落地分层

| 层级 | 关注点 |
|---|---|
| 知识源 | 文档、代码、接口、工单、日志、会议纪要等是否可信 |
| 解析层 | 文件格式、表格、图片、代码块、层级结构是否正确抽取 |
| 索引层 | chunk、metadata、embedding、关键词索引、增量更新 |
| 检索层 | 混合检索、权限过滤、rerank、去重、阈值 |
| 生成层 | 引用、拒答、冲突处理、格式约束 |
| 评估层 | 召回率、正确率、引用准确性、越权检测、成本延迟 |
| 运维层 | 监控、反馈闭环、版本回滚、知识更新 |

### 企业知识库重点

- 权限必须前置过滤，而不是生成后再遮盖。
- 文档要保留 owner、updated_at、业务域、系统名、版本号。
- 旧文档和新文档冲突时，要能识别更新时间和权威来源。
- 答案要带引用，否则无法复核。
- 对“知识库没有资料”的问题，要明确说明无法确认。

### 代码库 RAG 重点

- 普通文本切块不适合所有代码；代码应按类、函数、接口、配置块等结构切。
- 需要保留 repo、branch、path、symbol、language、commit 等 metadata。
- 问字段、接口路径、错误码时，关键词检索往往比纯向量更可靠。
- 代码答案最好引用文件路径、类名、方法名和行号。

### 评估问题

企业 RAG 至少要评估：

- 能否召回正确资料。
- 召回资料是否有权限。
- 答案是否忠于资料。
- 引用是否能定位到原文。
- 资料冲突时是否说明不确定。
- 无资料时是否拒答。
- 延迟和 token 成本是否可接受。

## 可执行动作

- [ ] 为企业知识库建立 metadata 字段清单。
- [ ] 建一份 RAG golden set，覆盖常见业务问法和边界情况。
- [ ] 把用户反馈分为“召回失败、重排失败、生成失败、知识缺失、权限问题”。
