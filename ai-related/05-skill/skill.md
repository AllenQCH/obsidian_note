# 1、skill是什么
Skill = 一个“结构化的能力模板”，本质是给 LLM 的可复用执行策略（prompt + 约束 + 工具用法）
AI Skill = 一个“边界清晰、输入输出明确、可被 Agent 调用的能力模块”
👉 “约束 LLM 如何使用工具解决问题”
MCP 是“工具协议”，Skill 是“业务能力”
Agent 决定“做什么”
Skill 决定“怎么做”
Tool / MCP 决定“怎么接系统”
这是 AI 系统工程里最稳的分层。
Skill.md ≠ 流程引擎  
Skill.md = Prompt（指导模型的文本）
没有“系统级 state machine 在驱动 skill”  
只有“LLM 在每一轮根据 prompt 决策下一步”

Skill 执行流程不是系统驱动的  
而是 LLM 在 loop 中“逐步推断”的  
  
所谓的“state”  
不是变量  
而是 context（历史 + 执行结果）  
  
跳步骤不是 bug  
而是模型在做“优化决策”

# 2、本质
👉 **Skill 是 Agent 的能力，不是 LLM 的能力**
**Skill 本质是 Prompt（不是程序）
```text
Skill.md = 一段被动态注入到 context 的结构化提示词
```
**Skill 不参与执行（没有 runtime）
```text
没有：  
- 状态机  
- 指令执行器  
- step 指针
```
Skill 的“流程”是如何起作用的？
```text
靠的是：LLM 在每一轮推理时，读取 Skill 描述，然后“推断下一步”
```
🔥核心公式：下一步行为 = f(当前context + skill描述 + 上一步结果)
# 3、skill原理
把“可执行能力”封装成一个黑盒，由 Agent 决策是否调用，Skill 负责确定性执行并返回结果。
基本工作循环
```text
开始
  ↓
接收任务输入
  ↓
┌─────────────────┐
│  输入验证        │ ← 检查参数有效性
└────────┬────────┘
         ↓
┌─────────────────┐
│  任务规划        │ ← 分解为子任务
└────────┬────────┘
         ↓
┌─────────────────┐
│  工具选择        │ ← 选择需要的工具
└────────┬────────┘
         ↓
┌─────────────────┐
│  顺序执行        │ ← 按顺序调用工具
│  - Tool 1       │
│  - Tool 2       │
│  - Tool N       │
└────────┬────────┘
         ↓
┌─────────────────┐
│  结果整合        │ ← 合并所有结果
└────────┬────────┘
         ↓
┌─────────────────┐
│  输出返回        │ ← 返回最终结果
└─────────────────┘
```

# 4、skill核心架构

Skill 在 Agent 架构中的位置
用户目标   ↓  
Agent（决策）   ↓选择 Skill  
   ↓  
Skill 执行   ↓结果返回 Agent
⚠️ Agent 不直接干活，只“选 Skill”。

# 5、skill真实执行机制
举例：一个skill如下：
```text
步骤：  
1. 查询数据  
2. 分析异常  
3. 生成SQL  
4. 执行  
5. 验证
```

# 6、Skill 的调用流程
下面是你想要的“从选中 → 执行完”的全过程。
## Step 0：启动时（索引阶段）
Codex 扫描：
~/.codex/skills/
生成：
Skill Index（只包含名称 + 简介）
👉 此时：
Skill 还没加载进 prompt
## Step 1：用户输入任务
“修复订单数据异常”
## Step 2：模型选择 Skill
模型看到：
- 用户输入  
- Skill Index
推理：
这个任务适合 sql-fixer
## Step 3：加载 Skill（关键）
系统做：
把 SKILL.md 内容注入 prompt
此时 context 变成：
```text
context =  
  用户问题  
+ history  
+ skill流程  
+ 工具说明
```
## Step 4：进入 Agent Loop（核心）
开始循环执行：
### 🔁 第1轮
模型思考：
根据 skill，第1步是查询数据
输出：
{ "tool": "db.query", "args": "SELECT ..." }
### Codex 执行 → 返回结果
### 🔁 第2轮
模型看到：
- 查询结果  
- skill流程
推理：
我已经做完第1步 → 现在应该做第2步（分析）
### 🔁 第3轮
生成 SQL
### 🔁 第4轮
执行 SQL
👉 一直到：
模型判断任务完成
# 7、真正执行时发生的事情（逐轮 loop）

### 第1轮（LLM推理）
模型看到：
```text
- 用户任务  
- skill步骤说明
```
模型内部推理：
```text
“我现在应该做第1步：查询数据”
```
👉 输出：
```text
{ "tool": "db.query", "args": "SELECT ..." }
```
**Codex执行 → 返回结果**
### 第2轮（LLM再次推理）
模型现在看到：
```text
- 上一步结果（查询结果）  
- skill步骤说明（还在）
```
模型推理：
“我已经查完数据了，现在应该做第2步：分析”
👉 输出：
```text
分析结果...
```
### 第3轮
```text
“下一步应该生成SQL”
```
👉 关键点来了：
```text
👉 “当前是第几步”不是系统记的  
👉 是模型自己“推断”的
```

# 8、state控制是怎么做的；
不是：
```text
系统内部：  
current_step = 2
```
而是：
```text
state = 上下文（context）
```
## 更具体一点：
模型的“状态”来自：
```text
context =  
  已执行步骤的结果（tool_result）  
+ 对话历史  
+ skill说明
```
👉 模型通过这些推断：
```text
我现在执行到哪一步了
```
## 为什么会出现跳步骤
```text
LLM 是“概率决策”，不是“严格执行器”
```
skill弱state导致跳步骤
LLM 是“概率决策”，不是“严格执行器”
### 举个真实情况
Skill 写：
```text
1. 查询  
2. 分析  
3. 生成SQL
```
但模型可能这样想：
```text
“我已经知道问题了，可以直接写SQL”
```
👉 于是：
```text
跳过步骤1和2
```
### 为什么会这样？
因为模型在优化：
```text
“用最少步骤完成任务”
```
👉 所以：
```text
Skill 的“步骤”只是 guidance（指导），不是强约束
```
# 9、skill的弱state控制
来自这三样：
```text
1. tool_result（刚刚执行了什么）  
2. history（之前做了什么）  
3. skill文本（流程定义）
```
模型综合判断：
```text
“我已经做了A，现在应该做B”
```

## 什么是强state

```text
Step1: 必须执行  
Step2: 必须执行  
Step3: 必须执行
每一步都写得非常明确 + 强约束
```
### 强state方案：
#### 方案1：Agent（推荐）
👉 把流程写死

#### 方案2：外部控制（更强）
例如：
```text
你写Java代码：  
step1()  
step2()  
step3()
每一步调用 Codex
```
