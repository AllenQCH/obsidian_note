# 网页自动化方案
> 1、**接口层**：直接发 HTTP 请求，最稳、最快
> 2、**页面层**：操作 DOM（**DOM = 浏览器把 HTML 变成“可操作的对象树”**），点按钮、填表单、读页面内容
> 3、**浏览器层**：直接控制 Chrome，本质像“真人操作浏览器”

## 方案一：纯接口方案
你的流程其实主要是“提交数据、查询状态、下载结果”，页面只是个壳。
先在浏览器 Network 里抓包，找到真正调用的接口，然后自己写程序直接请求这些接口。  
本质是绕过前端页面，直接调用后端服务。
### 优点
- 最快、最稳定、成本最低、不怕页面 DOM 改版
### 缺点
- 你得搞清楚鉴权、cookie、header、token、签名
- 有些系统前端校验很多，单发接口会失败
- 有些流程依赖页面内 JS 状态，不好复现
因为很多业务系统不是简单 REST API，而是：
- 登录态绑定浏览器
- token 动态刷新
- 页面先调 A 再调 B，再拼出 C
- 某些字段由前端 JS 算出来
- 操作前需要先拿到隐藏表单值或页面上下文
所以“调用接口”不是错，而是**经常只能覆盖 60%-80% 的流程**。剩下的要靠页面自动化补齐。

## 方案二：Chrome 插件，一键执行固定流程
### 适合
你的流程很固定，比如：
- 打开某页面；找几个输入框填值；点提交；等结果；导出数据
### 原理
Chrome 插件一般是这几个部分协作：
- **content script**：注入页面，直接读写 DOM
- **service worker / background**：负责全局逻辑
- **message passing**：插件各部分互相传消息
- **chrome.scripting**：按需把脚本注入指定页面
Chrome 官方文档里明确说了：  
content script 能访问页面 DOM（**DOM = 浏览器把 HTML 变成“可操作的对象树”**），但很多扩展 API 需要通过消息机制和其他扩展部分配合；`chrome.scripting` 可以在运行时注入 JS/CSS。
### 优点  
- 对用户最友好，点一下就能跑  
- 复用当前浏览器登录态  
- 很适合内部工具  
- 不一定要接 LLM  
### 缺点  
- 页面改版后，DOM 选择器容易失效  
- 跨 iframe、shadow DOM、复杂 SPA 时会变麻烦  
- 适合固定流程，不适合大量推理判断  
### 最适合你的场景  
如果你说的是“我就想把某个网页流程变成一个内部小工具，自己或同事一键跑”，    
**Chrome 插件通常是第一选择**。  

## 方案三：Playwright / Puppeteer 浏览器自动化  
### 适合  
你想做：  
- 自动登录/复用会话  
- 跨页面流程  
- 更稳定的点击/等待/读取  
- 后续还能接 agent  
### 原理  
直接让程序控制浏览器。    
Playwright 官方文档把它定位为可用于“programmatic browser control”的库，支持点击、输入、选择、上传等页面动作；它的 locator 机制和 auto-waiting 机制就是为“页面不稳定、异步加载”这种情况准备的。
Playwright 官方建议优先用用户可见语义定位，比如 `getByRole()`。这类定位比纯 CSS class 更抗改版。
### 缺点  
- 需要单独运行自动化服务  
- 部署比插件麻烦  
- 不是“天然一键给业务同学用”的形态  
### 适合怎么做  
你可以先把流程写成 Playwright 脚本，再做两层封装：  
- 给人用：做成命令行 / 小页面 / 内部平台按钮  
- 给 agent 用：封装成 skill / tool  
## 方案四：Agent Skill + 浏览器工具  
### 适合  
你的流程不是死流程，而是有分支判断，比如：  
- 看到状态 A 就点确认  
- 看到状态 B 就回退重新填写  
- 如果页面报错就重试  
- 需要读页面文案后决定下一步  
### 原理  
把浏览器操作暴露成一组工具，例如：  
- `open_page(url)`  
- `click(selector)`  
- `type(selector, text)`  
- `extract_text(selector)`  
- `wait_for(text)`  
- `screenshot()`  
然后 agent 根据 skill 的步骤去调用这些工具。    
也就是说：  
**LLM 负责“想下一步做什么”**    
**浏览器自动化底座负责“真的去点、去填、去读”**  
### Skill 怎么写  
Skill 不应该写成“思维提示词”，而应该写成：  
- 目标  
- 前置条件  
- 步骤模板  
- 分支规则  
- 失败处理  
- 输出格式   
例如：  
目标：在订单页面完成审核提交流程  
前置条件：  
- 用户已登录系统  
- 当前在订单详情页  
步骤：  
1. 读取订单状态  
2. 若状态=待审核，点击“审核”  
3. 填写备注  
4. 点击“提交”  
5. 等待出现“成功”提示  
6. 返回订单号与提交时间  
失败处理：  
- 若出现“权限不足”，终止并返回错误码  
- 若 10 秒内按钮不可点击，刷新后重试一次
### 优点
- 灵活
- 可处理半结构化页面
- 后续能扩展更多流程
### 缺点
- 成本比固定插件高
- 稳定性取决于 skill 设计 + 工具质量
- 没有良好的 guardrail 时容易“乱点”
### 适合你什么时候选
你要的是“不是一个按钮跑到底，而是让它像个助手一样会判断”，就走这条。

## 方案五：OpenCLI 路线
### 适合
你既想要：
- 复用浏览器登录态
- 不想自己从零写整套浏览器控制
- 又想让 agent 很容易调用
### 它的原理
我查到现在比较热的这个 OpenCLI（jackwener/opencli）主打的是：
- 通过 **Chrome/Chromium 的 Browser Bridge 扩展**
- 配合一个本地 daemon
- 直接复用当前 Chrome 的登录态
- 提供浏览器控制能力
- 还能把网站“包装成 CLI 命令”
- 有 `explore / synthesize / browser / cascade` 这类能力，用于发现 API、生成适配器、直接浏览器操作等。“测试框架”，也不是传统 Chrome 插件。  
    更像是：

**把网页 / Electron 应用 / 本地工具，包装成 agent 可调用的 CLI 能力层。**
所以它很适合做这种事：
- 人先在 Chrome 登录业务系统
- OpenCLI 复用这个已登录会话
- 你把“网页操作流程”抽成一组命令
- agent 再去调用这些命令
### 它对你有什么价值
如果你不想自己从头处理：
- 浏览器连接
- 会话复用
- 页面控制接口包装
- 给 agent 暴露统一命令入口
那 OpenCLI 确实能简化一部分工作。“连接浏览器 + 暴露操作能力 + 复用登录态”。  
它**不能替你定义业务流程本身**。  
你仍然要自己解决：
- 哪些页面元素最稳定
- 哪些步骤需要判断
- 什么叫成功/失败
- 页面改版后怎么维护
所以它更像“省掉底座”，不是“省掉设计”。
