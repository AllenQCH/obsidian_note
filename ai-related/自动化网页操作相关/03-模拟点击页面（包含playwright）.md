模拟点击页面”本质上就是：  
**用程序去做“人用鼠标键盘做的事”**，而不是你手动去点。
程序操作（模拟点击）：
- 打开网页 → **代码触发点击事件** → **代码填输入框** → **代码提交**

# 拆开讲三种“模拟点击”的层次

## 1、最简单：操作 DOM（前端层）

这是你最常听到的“模拟点击”。
### 原理
网页里的按钮本质是一个 DOM 元素，比如：
<button id="submit">提交</button>
你可以用 JS 直接触发：
document.querySelector('#submit').click()
或者：
element.dispatchEvent(new MouseEvent('click'))
👉 这就是“模拟点击”——**不是真的点鼠标，而是触发点击事件**

## 2、浏览器自动化（更真实）

比如用 Playwright / Puppeteer。
### 原理

不是简单调用 `.click()`，而是：
- 找到元素
- 滚动到可见区域
- 检查是否可点击
- 再执行点击
类似：
await page.click('#submit')
它内部其实做了：
- 等待页面加载
- 等待按钮可见
- 再触发点击
👉 相当于“高级模拟点击”，更接近真人操作

## 3、 真·模拟用户行为（底层）
比如通过浏览器协议（CDP）或系统事件：
- 移动鼠标到坐标
- 按下鼠标
- 松开鼠标
这种更像：
move mouse → down → up
👉 用于：
- 反爬场景
- 验证码绕过
- 游戏自动化

# 和“调用接口”的区别（关键点）
你之前的疑惑就在这里：
## 调接口
直接请求后端 API
## 模拟点击
走前端页面逻辑 → 页面再调接口

# 举个你这个场景的例子（加班调休）
你手工操作是：
1. 打开页面
2. 点“新增工单”
3. 填时间
4. 点提交
## 调接口方式
POST /createWorkOrder  
{  
  startTime: xxx,  
  endTime: xxx  
}

## 模拟点击方式
程序做的是：
1. 打开网页  
2. 找到“新增按钮” → click()  
3. 找到输入框 → 填值  
4. 找到提交按钮 → click()
# 为什么大家会用“模拟点击”
因为很多系统：
### ❌ 不能直接调接口
原因包括：
- token 动态生成
- cookie 绑定浏览器
- 请求有签名
- 需要先加载页面数据
- 前端拼装参数

👉 你直接调接口会失败

### ✅ 模拟点击就能绕过
因为：
- 你复用了浏览器登录态
- 页面帮你做了所有校验
- JS 帮你生成参数
👉 相当于“借用前端逻辑”

# 用一句更工程化的话总结
**模拟点击 = 在浏览器上下文里执行 UI 操作，从而间接触发后端请求**
# 再帮你纠正一个常见误区
很多人以为：
> 模拟点击 = 用代码点一下按钮
其实更完整的是：
👉 **一整套 UI 自动化行为**  
包括：
- 点击
- 输入
- 选择下拉框
- 等待加载
- 判断页面状态
- 读取结果

回到你“加班调休”的例子：
如果你用模拟点击，流程会变成：
程序：  
1. 打开调休页面  
2. 点击“新建”

4. 提交  
5. 再点“新建”  
6. 填第二张单  
7. 提交



**Playwright** 是一个浏览器自动化库（微软出品），用来**用代码控制浏览器行为**。
它不是在“网页外面操作”，而是在“浏览器内部控制浏览器”

浏览器其实提供了一套“远程控制协议”（例如 Chrome DevTools Protocol）：
你（程序） → 浏览器 → 页面
Playwright 做的事情是：
1. 启动一个浏览器实例
2. 通过协议连接这个浏览器
3. 发指令：
点击这个按钮  
输入这个内容  
获取这个元素
浏览器接到指令后，**真的去执行这些动作**。

### 举个对比
你手动：
鼠标点按钮
Playwright：
await page.click('#submit')
本质是：
程序告诉浏览器：帮我点一下这个按钮
👉 所以它不是“假点击”，而是**浏览器帮你点了**

## Playwright 和“普通 JS 点击”的区别

|方式|本质|稳定性|
|---|---|---|
|`element.click()`|直接触发事件|❌ 很容易失败|
|Playwright `page.click()`|浏览器完整执行点击|✅ 很稳|

### Playwright 为什么更稳
它内部帮你做了很多事：
- 等元素出现
- 等元素可见
- 滚动到可见区域
- 确保不被遮挡
- 再点击
👉 相当于：**模拟一个“正常用户行为”**
