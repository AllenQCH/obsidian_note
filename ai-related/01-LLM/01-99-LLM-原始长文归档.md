# 01-99-LLM-原始长文归档

> 这份文件保留原始长文和迁移痕迹。正式阅读入口见 [[01-01-LLM-总览]]；基础主题见 [[01-02-LLM-Token]]、[[01-03-LLM-Prompt]]、[[01-04-LLM-Context]]、[[01-05-LLM-Temperature]]、[[01-06-LLM-Memory]]；RAG 体系见 [[02-01-RAG-总览]]。

## 摘要

- LLM 是负责理解、推理、决策和生成文本的通用推理引擎。
- 只依赖 LLM 远远不够：`RAG` 给知识，`Tool` 给执行能力，`Memory` 保持状态，`Agent` 组织多步任务。
- LLM 本身没有 IO、执行环境和长期记忆，也不能保证结果 100% 正确。

## 核心内容

### 为什么只要 LLM 不够

| 补充组件 | 解决的问题 | 相关笔记 |
|---|---|---|
| RAG | 给 LLM 外部知识 | [[02-01-RAG-总览]] |
| Tool | 让 LLM 能调用外部能力 | [[Tool]] |
| Memory | 保存状态和经验 | [[01-06-LLM-Memory]] |
| Agent | 组织多步任务和执行循环 | [[00-agent introduction]] |
| MCP | 标准化工具和上下文连接 | [[mcp]] |

### LLM 擅长什么

| 能力 | 例子 |
|---|---|
| 语义理解 | 解释一段代码在做什么 |
| 推理 | 判断某一步失败后下一步怎么做 |
| 决策 | 判断是否需要调用某个工具 |
| 生成 | 生成 SQL、Java、文档、Prompt |

### LLM 不擅长什么

| 做不到 | 原因 |
|---|---|
| 查实时数据库 | 没有 IO |
| 调用接口 | 没有执行环境 |
| 记住长期历史 | 上下文是一次性的 |
| 处理超大数据 | 有 token 上限 |
| 保证 100% 正确 | 是概率模型 |

### LLM 是什么

LLM 是一个“只会思考 + 生成文本”的通用推理引擎。它依赖两类信息：

- 它原本学过什么，也就是训练数据。
- 你临时在对话框里告诉它什么，也就是 [[01-03-LLM-Prompt]]、[[01-04-LLM-Context]]、工具结果和记忆。

### 在 AI Agent 架构中的位置

```text
LLM：负责想
Agent：负责决策流程
Tool / Skill：负责做事
DB / API：真实外部系统
```

### 模型选择原则

- 代码任务看代码质量、上下文长度、工具调用稳定性。
- 复杂推理看长链路推理和自我纠错能力。
- 日常对话看速度和成本。
- 快速原型看交互速度和工程集成成本。

原 OneNote 笔记中涉及具体模型版本和价格，这类信息具有强时效性，需要单独核验。






你这个是典型的大模型推理参数配置页，基本上所有 Agent 平台（Dify、百炼、Coze、OpenAI Playground）都会有类似概念。  
这些参数本质上是在控制：

> “模型怎么思考、思考多久、输出有多随机、输出格式是什么、是否联网”

# 2. temperature（温度）

你现在：

```text
temperature = 0.10
```

这个参数：

> 控制“随机性”

---

## 温度低（0~0.3）

特点：

- 更稳定
    
- 更严谨
    
- 更像固定答案
    
- 更少发散
    
- 更适合：
    
    - 写代码
        
    - SQL
        
    - Agent执行
        
    - 工作流
        
    - 生产环境
        

例如：

问：

```text
Java 判断空字符串
```

temperature=0.1：

```java
StringUtils.isBlank(str)
```

非常稳定。

---

## 温度高（0.8~1.5）

特点：

- 更发散
    
- 更有创造力
    
- 更像 brainstorming
    
- 更容易“胡说”
    

适合：

- 写小说
    
- 起名字
    
- 文案
    
- 创意内容
    

---

# 第一性原理

本质上：

模型下一步会预测很多 token：

```text
[ tokenA: 70%
  tokenB: 20%
  tokenC: 10% ]
```

temperature 越高：

概率越被“拉平”。

模型越敢选不常见 token。

# 3. top_p

你现在：

```text
top_p = 0.80
```

这是：

> “从多少概率范围内挑词”

叫：

```text
Nucleus Sampling（核采样）
```

---

## 举例

模型预测：

|token|概率|
|---|---|
|A|50%|
|B|20%|
|C|15%|
|D|10%|
|E|5%|

---

## top_p = 0.8

模型会：

```text
A + B + C = 85%
```

到这里够了。

后面的 D/E 直接丢弃。

于是：

模型只能从：

```text
A/B/C
```

里面选。

---

# temperature vs top_p

很多人容易混。

实际上：

|参数|控制什么|
|---|---|
|temperature|“敢不敢乱选”|
|top_p|“允许在哪些候选里选”|

---

# 实际经验

## Agent / Coding

推荐：

```text
temperature = 0~0.3
top_p = 0.7~0.9
```

因为：

Agent 最怕：

- 发散
    
- 幻觉
    
- 自作聪明
    

---

## 创作

推荐：

```text
temperature = 0.8+
top_p = 0.95+
```

# 4. enable_thinking

你这里是：

```text
关闭
```

这是现在最核心的新参数。

它决定：

> 模型是否进入 reasoning / chain-of-thought 模式

也就是：

```text
“会不会先思考，再回答”
```

---

# 关闭 thinking

模型：

```text
输入 -> 直接输出
```

特点：

- 快
    
- 便宜
    
- 延迟低
    
- 适合简单问答
    

---

# 开启 thinking

模型：

```text
输入
  ↓
内部推理
  ↓
规划
  ↓
检查
  ↓
输出
```

特点：

- 更强推理
    
- 更适合：
    
    - 数学
        
    - Agent
        
    - 复杂代码
        
    - 多步骤任务
        
    - Debug
        
    - 架构设计
        

代价：

- 更慢
    
- 更贵
    
- token 消耗更大
    

---

# 本质区别

以前 GPT：

```text
next token prediction
```

现在 reasoning model：

```text
先生成 hidden reasoning token
再生成 answer token
```

这就是：

```text
test-time scaling
```

也就是：

> “让模型多想一会”

# enable_search这个开关，它打开与否；它是在llm之前还是之后呢？如果有一个搜索，也会用到RAG；一直到返回，是个什么流程；
这个问题已经进入 Agent / Search-Augmented LLM 的核心了。

简单说：

> `enable_search` 不是“LLM内部能力”  
> 而是：
> 
> ```text
> Runtime / Agent Framework 给 LLM 增加的外部能力
> ```

所以它既不是“之前”，也不是“之后”。

而是：

```text
LLM 推理过程中
动态调用 Search Tool
```

更准确。

# 二、真正完整流程（核心）

现代 AI Search 基本都是：

```text
用户问题
   ↓
LLM 判断是否需要搜索
   ↓
调用 Search Tool
   ↓
搜索引擎检索
   ↓
返回网页 / snippets
   ↓
RAG（检索增强）
   ↓
LLM 阅读搜索结果
   ↓
总结生成最终答案
```

真正结构：

```text
           ┌──────────────┐
           │ User Prompt  │
           └──────┬───────┘
                  ↓
        ┌──────────────────┐
        │   LLM Runtime    │
        │  (Agent Brain)   │
        └──────┬───────────┘
               │
     decide whether search needed
               │
        ┌──────▼─────────┐
        │ Search Tool/API │
        └──────┬─────────┘
               ↓
      Google/Bing/内部搜索
               ↓
      返回网页/摘要/snippets
               ↓
        ┌──────────────┐
        │ RAG Pipeline │
        └──────┬───────┘
               ↓
     relevant chunks/context
               ↓
        ┌──────────────┐
        │ LLM 再推理    │
        └──────┬───────┘
               ↓
           最终答案
```

# 四、从第一性原理理解

LLM 本质：

```text
next token prediction
```

它：

- 不会联网
    
- 不会主动获取知识
    
- 不会自己HTTP请求
    

所以：

```text
enable_search
```

实际上是：

```text

```

类似：

```text
tools = [
  search(),
  browser(),
  calculator(),
  code_interpreter()
]
```

LLM 在推理中决定：

```text
我需要调用哪个工具
```

# 五、现代 enable_search 的真实内部

通常分三层：

---

```text

```

而是先：

```text

```

```text

```

这是：

```text
Search Planning
```

---

# 第二层：Search Retrieval

搜索：

```text
Google/Bing/内部向量库
```

返回：

```text
title
snippet
url
content
```

例如：

```json
[
  {
    "title": "...",
    "snippet": "...",
    "url": "..."
  }
]
```

---

# 第三层：RAG

重点来了。

搜索结果：

不是直接给用户。

而是：

```text
拼接进 context
```

变成：

```text
[System Prompt]

[User Question]

[Retrieved Search Results]

...

Result2:
...

请基于这些结果回答。
```

这一步：

就是：

# RAG（Retrieval-Augmented Generation）

相当于：

```text
给 Agent 一个浏览器
```

---

## 本地RAG

相当于：

```text
给 Agent 一个长期记忆库
```

---

# 六、为什么很多 Agent Demo 不做 RAG

因为：

真正复杂的是：

```text
知识库治理
```

不是 embedding。

---

真正难点：

- 文档同步
    
- chunk切分
    
- 权限
    
- 更新
    
- 去重
    
- metadata
    
- rerank
    
- 多版本
    
- 召回质量
    
- 幻觉控制
    

所以：

很多产品早期：

```text
先做 search
```

已经能解决：

```text
80% 通用问题
```

---

# 七、你现在看到的大多数 AI 产品

其实是：

```text
Search-Augmented LLM
```

不是完整企业RAG。

例如：

|产品|本质|
|---|---|
|ChatGPT联网|Search + LLM|
|Perplexity|Search + RAG + LLM|
|Gemini Search|Search Agent|
|Claude Web Search|Search Tool|

---

# 八、真正企业级 Agent

才会：

```text
Search
+
Internal RAG
+
Memory
+
Workflow
+
Tool Calling
```

例如：

```text
客服
财务
运维
代码助手
```

---

# 九、所以你现在应该这样理解

---

# Search

解决：

```text
“模型不知道最新信息”
```

---

# RAG

解决：

```text
“模型不知道你的私有知识”
```

---

# Memory

解决：

```text
“模型不记得历史”
```

---

# Agent Workflow

解决：

```text
“模型不会稳定执行复杂流程”
```

---

# 十、最重要一句

现代 AI 系统：

```text
LLM 已经不是核心本体了
```

而是：

```text
LLM + Retrieval + Memory + Tool + Workflow
```

共同构成：

# AI Runtime / Agent Runtime

这也是为什么现在大家越来越少单独讨论“模型”，而开始讨论：

- Agent
    
- Runtime
    
- Orchestration
    
- Context Engineering
    
- Tool System
    
- Memory System
    

因为：

```text
单纯模型能力已经开始接近瓶颈
```

真正拉开差距的：

是外围系统。
