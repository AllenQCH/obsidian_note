# 第一部分：知识上传后的全过程（离线）
# Step 1：上传文档

可能上传：
- PDF
- Word
- Excel
- Markdown
- Wiki
- 飞书
- Confluence
- GitLab
- MySQL
- Jira
例如：
```text
退款系统设计.pdf
```
# Step 2：文档解析（Parser）
这里是第一大坑。
因为：
## LLM 吃的是文本
但 PDF 不是纯文本。
PDF 实际可能：
```text
文字 + 图片 + 表格 + OCR
```
所以必须：
```text
PDF → 结构化文本
```
## 不同文档的解析器

|文件|常见解析|
|---|---|
|PDF|OCR / Layout Parser|
|Word|docx parser|
|Excel|sheet parser|
|HTML|DOM parser|
|Markdown|markdown parser|

## 真正工业级解析很复杂
比如：
## PDF 中：
```text
左边是正文
右边是注释
底下是页脚
```

如果解析错：
LLM 就会看到：
```text
退款接口页脚Redis连接池章节页码
```
直接废掉。

## 所以现在很多 RAG 公司：
真正壁垒其实是：
## “文档解析能力”
不是 LLM。

# Step 3：文档清洗（Cleaning）
解析后内容很脏：
比如：
```text
页码
广告
乱码
重复标题
页脚
导航栏
```
都得删。
## 为什么？
因为：
## Garbage In Garbage Out
如果脏数据进 Prompt：
LLM 会胡说。

# Step 4：Chunk 切块（核心）
这是：
## RAG 最核心步骤之一
因为：
LLM Context 有限。
不能整个 PDF 塞进去。
所以：
```text
长文档
↓
切成小块
```
# Chunk 为什么重要
因为：
# “检索单位就是 Chunk”
不是文档。
# 举例
原文：
```text
第一章：订单系统

退款接口超时原因：
1.Redis连接池耗尽
2.MQ阻塞
3.数据库死锁
```
切成：
```text
chunk_001:
第一章订单系统

chunk_002:
退款接口超时原因...
```
## Chunk 太大问题
如果：
```text
5000字一个chunk
```
问题：
- embedding 不精准
- topic 太混
- 检索污染
- prompt太长
## Chunk 太小问题
如果：
```text
一句话一个chunk
```
问题：
- 上下文丢失
- 语义断裂
- 回答碎片化

## 所以 Chunk 是门艺术
工业里常见：

|类型|说明|
|---|---|
|Fixed Chunk|固定长度|
|Sliding Window|滑动窗口|
|Semantic Chunk|按语义切|
|Hierarchical Chunk|父子分层|
## 最经典：

### Sliding Window
比如：
```text
chunk1:
1~500字

chunk2:
400~900字
```
会有 overlap。
## 为什么 overlap
防止：
```text
一句话被切断
```
导致语义丢失。
# Step 5：Metadata 元数据
每个 chunk 不只有文本。
还会带：
```json
{
  "doc_id": "refund_design.pdf",
  "page": 12,
  "title": "退款超时",
  "department": "支付中心",
  "create_time": "2026-01-01"
}
```
## 为什么 metadata 很重要
因为后面检索：
不仅是语义检索。
还会：
```text
过滤
排序
权限控制
时间过滤
部门过滤
```
# Step 6：Embedding 向量化
真正进入 AI 世界。
## Embedding 本质
把文本：
```text
退款失败
```
变成：
```text
[0.123, 0.555, 0.888...]
```
高维向量。
## 为什么要向量
因为：
计算机不懂语义。
但：
## 向量距离 ≈ 语义距离

## 举例
```text
退款失败
支付超时
```
两个向量会很近。
而：
```text
退款失败
今天下雨
```
会很远。

## Embedding Model
常见：
- OpenAI text-embedding
- Cohere rerank/embed
- BAAI bge
- Jina AI embeddings
# Step 7：存入 Vector DB
最后：
```text
chunk + vector + metadata
```
存进去。
例如：
```json
{
  "text": "退款接口超时原因...",
  "vector": [0.123, 0.555...],
  "metadata": {...}
}
```

## 向量数据库干什么
核心：
## 相似度搜索
## 常见 Vector DB
- Milvus
- Pinecone
- Weaviate
- Qdrant
- Elasticsearch
## 到这里：
## “知识库建好了”
## 一句话总结整个流程
## 离线：
```text
文档 → chunk → embedding → vectorDB
```
## 在线：
```text
问题 → 检索 → rerank → prompt → LLM
```
而真正核心：
## “让正确上下文，在正确时间进入 Prompt。”
