

# 第二部分：用户提问后的全过程（在线）
# Step 8：用户提问
用户：
```text
退款为什么超时？
```
# Step 9：Query Rewrite（高级RAG）
很多系统不会直接搜。
而是：
## 先改写问题
比如：
用户：
```text
为什么这么慢？
```
系统会改写：
```text
退款接口响应慢原因
```
## 为什么？
因为：
用户问题经常：
- 模糊
- 口语化
- 缺上下文
# Step 10：Query Embedding
用户问题也向量化。
变成：
```text
query vector
```
# Step 11：向量检索 Retrieval

然后：

```text
query vector
vs
所有chunk vector
```

做相似度搜索。

## 常见算法

|算法|作用|
|---|---|
|Cosine|余弦相似度|
|L2|欧氏距离|
|ANN|近似最近邻|

## 为什么 ANN
因为：
企业可能：
```text
1亿 chunk
```
不能暴力算。
所以：
要近似搜索。



## 最终召回：

```text
Top K chunks
```

比如：

```text
chunk_22
chunk_89
chunk_104
```



# Step 12：Rerank（工业级关键）

很多人以为：

检索完就结束。

其实不是。

因为：

## 向量检索经常不准



## 所以需要：

## 重排序模型（Rerank）

它会：

```text
query + chunk
逐个精排
```

---

## 和 embedding 区别

Embedding：
```text
粗召回
```
Rerank：
```text
精排序
```

## 类似：

### 搜索引擎：

```text
先海选
再精排
```



# Step 13：Context Compression

有时 chunk 太多。
还会：
- 去重    
- 摘要
- 压缩    
否则：
Prompt 太长。

# Step 14：Prompt Assembly（最关键）
现在：
终于开始组 Prompt。

## 真正 Prompt 长这样

```text
System Prompt:
你是退款系统专家...

Context:
[chunk1]
退款超时原因...

[chunk2]
Redis连接池说明...

Question:
退款为什么超时？
```



## 你终于看到：

## RAG 真正本质：

```text
“动态给LLM外挂上下文”
```



# Step 15：LLM Generation

最后：
LLM 才开始生成。
## 注意：
LLM 并不知道：
```text
什么是知识库
什么是向量数据库
什么是RAG
```
它只看到：
```text
Prompt里多了一堆上下文
```


## 工业级 RAG 还会继续增强

现代高级 RAG：



## Hybrid Search（混合检索）

同时：

- BM25关键词
    
- 向量检索
    

避免：

语义漂移。



## GraphRAG

不仅检索 chunk。

还检索：

```text
实体关系图
```

比如：

```text
订单 -> 店铺 -> 仓库 -> 发票
```



## Agentic RAG

Agent 自己：

- 决定搜什么
    
- 多轮搜索
    
- 反思
    
- 迭代检索
    



## Long Context RAG

甚至：

不用切 chunk。

直接：

```text
整个仓库塞给超长上下文模型
```

比如：

- Google Gemini
    
- Anthropic Claude
