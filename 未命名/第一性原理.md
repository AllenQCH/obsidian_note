# 一、什么是第一性原理
> 把问题拆到不可再分的基础事实（第一性），再从这些事实重新推导结论。
它的对立面是：  
**类比思维 / 经验思维**（别人这么做 → 我也这么做）
## 一个对比就清楚
### 问题：为什么要用缓存？
**经验思维：**
- 大家都用 Redis → 我也用
**第一性原理：**
- CPU快，内存快，磁盘慢
- 网络访问慢于本地访问  
    → 推导：把热点数据放内存可以降低延迟
👉 这才是“理解”，不是“记住”

# 二、哪些事情适合用第一性原理
不是所有问题都值得用，否则效率会很低。
## 1）非常适合用的场景

### （1）复杂系统 / 分布式系统

例如你日常工作：
- 分库分表
- MQ
- ES
👉 特点：  
规则复杂、边界多、异常多
👉 为什么适合：  
经验很容易失效，只能靠本质推导
### （2）新领域学习（你现在的AI转型）
- Agent是什么
- Tool Calling / MCP
- LLM为什么会幻觉
👉 如果只记概念，会很碎  
👉 用第一性原理可以“搭骨架”
### （3）没有标准答案的问题
- 架构设计
- 性能优化
- 业务重构
👉 这些问题本质是“推导题”，不是“记忆题”
### （4）你觉得“别人说的不太对”的时候
👉 这是最重要的使用时机
## 2）不适合用的场景
### ❌ 标准化工作
- CRUD
- 调接口
- 用成熟框架
👉 直接用最佳实践更高效
### ❌ 时间紧的场景
- 线上紧急修bug
👉 先解决，再复盘用第一性原理
### ❌ 已经高度工程化的问题
- Spring配置
- MyBatis写法
👉 这些已经被“封装过了”
# 三、如何使用第一性原理（核心方法）
给你一套可以直接用的**四步法**：
## Step 1：拆问题（拆到最小单位）
目标：把问题拆到“不能再拆”
### 示例：缓存一致性
拆成：
- 数据副本（DB + Cache）
- 操作顺序（先写谁）
- 并发请求
- 网络延迟
👉 注意：  
不要停在“模块级”，要拆到“行为级”
## Step 2：找“不可改变的约束”
这是核心
### 常见约束（你这类系统）
- 网络一定会延迟
- 系统一定会失败
- 分布式没有全局时钟
- 并发一定存在
👉 这些是“物理规律级别”，不是业务规则
## Step 3：从约束推结论
这是最关键的一步
### 还是缓存例子
有这些约束：
- 写操作不是原子
- 请求可能乱序
👉 推导：
**缓存不一致是必然的**
👉 这一步会让你“认知跃迁”
## Step 4：再推方案（不是记方案）
从结论反推：
- 既然一定不一致 → 做最终一致性
- 既然有乱序 → 延迟双删
- 既然会失败 → MQ补偿
👉 这时你就不是“用方案”，而是“生成方案”
# 四、给你一个通用分析模板（可以直接用）
你以后可以这样分析任何问题：
### 模板：
1️⃣ 这个问题的最小组成是什么？  
（拆）
2️⃣ 每个组成部分的“基本行为”是什么？  
（理解机制）
3️⃣ 有哪些不可避免的约束？  
（物理/系统规律）
4️⃣ 在这些约束下，必然会发生什么？  
（推导结论）
5️⃣ 我们现在的方案，本质是在解决哪一个约束？  
（反推方案）

# 五、一个完整示例（AI领域）

## 问题：为什么需要 Agent？

---

### Step 1：拆

- LLM
- 工具（API）
- 任务流程

---

### Step 2：约束

- LLM只能生成文本
- 无法直接操作系统
- 无法长期记忆
- 无法自主执行流程

---

### Step 3：推导

👉 LLM本身无法完成复杂任务

---

### Step 4：方案

👉 Agent = LLM + Tool + Memory + Orchestration

---

👉 这就是为什么 Agent 会火（不是趋势问题，是必然结果）

---

# 六、总结一句话

**第一性原理 = 从“世界怎么运作”出发，而不是“别人怎么做”出发**
